# V4.2.0 — RedisCacheAdapter

## Visão Geral

Esta etapa introduz o adapter responsável pela implementação concreta da porta de cache utilizando Redis.

O adapter encapsula completamente os detalhes de infraestrutura relacionados ao Redis preservando o isolamento arquitetural da aplicação.

---

# O que foi implementado

## RedisCacheAdapter

Foi criada a implementação da porta `TranscriptionCachePort`.

### Responsabilidades

| Responsabilidade | Objetivo |
|---|---|
| Salvar cache | Reutilização futura |
| Buscar por hash | Deduplicação |
| Aplicar TTL | Expiração automática |
| Padronizar chave | Consistência |

---

## Estratégia de chave

Formato adotado:

```text
transcription:{hash}
```

### Benefícios

- Organização lógica
- Facilidade operacional
- Busca previsível
- Compatibilidade futura

---

## TTL configurável

O tempo de expiração é definido via:

```yaml
app.cache.transcription-ttl-hours
```

---

## Estratégia de profiles

### Produção

```java
@Primary
RedisCacheAdapter
```

### Testes

```java
@Profile("test")
InMemoryCacheAdapter
```

---

# Decisão de arquitetura

## Por que adapter separado?

O domínio não deve conhecer Redis diretamente.

O adapter preserva:

- Hexagonal architecture
- Substituição futura
- Testabilidade
- Isolamento técnico

---

## Por que manter InMemory nos testes?

Evita dependência de infraestrutura externa em testes unitários.

---

# Resultado da etapa

A aplicação agora possui:

- Cache distribuído Redis
- Adapter desacoplado
- TTL configurável
- Profiles separados
- Estratégia consistente de chave
