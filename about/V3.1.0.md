# v3.1.0 — Setup PostgreSQL + JPA + Flyway

## Visão Geral

Esta etapa introduz a infraestrutura inicial de persistência da aplicação utilizando PostgreSQL, Spring Data JPA e Flyway.

O objetivo desta implementação é preparar a base estrutural necessária para futuras funcionalidades de persistência de transcrições, deduplicação por hash, cache e integração assíncrona, mantendo a arquitetura hexagonal desacoplada da infraestrutura.

A camada de domínio permanece completamente agnóstica ao banco de dados e ao framework ORM.

---

# O que foi implementado

## PostgreSQL no ambiente Docker

Foi adicionada uma instância PostgreSQL ao ambiente local utilizando Docker Compose com persistência em volume.

### Responsabilidades

| Componente | Responsabilidade |
|---|---|
| postgres container | Banco relacional principal |
| volume persistente | Preservação de dados entre reinicializações |
| variáveis de ambiente | Configuração do banco |

### Trecho principal

```yaml
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_DB: speech_ai
    volumes:
      - postgres_data:/var/lib/postgresql/data
```

---

## Spring Data JPA

Foi configurada a infraestrutura ORM utilizando Spring Data JPA.

### Objetivos

- Padronizar acesso futuro à persistência
- Preparar repositories desacoplados
- Permitir adapters de saída isolados
- Facilitar evolução futura do bounded context `transcription`

### Dependências adicionadas

| Dependência | Finalidade |
|---|---|
| spring-boot-starter-data-jpa | Infraestrutura JPA/Hibernate |
| postgresql | Driver JDBC PostgreSQL |
| flyway-core | Versionamento de schema |

---

## Flyway

Foi introduzido Flyway para controle versionado de migrations SQL.

### Benefícios

- Histórico rastreável de alterações
- Ambiente reproduzível
- Controle explícito de schema
- Segurança em evolução incremental

### Estrutura esperada

```text
src/main/resources/db/migration
```

### Convenção de migrations

```text
V1__initial_schema.sql
V2__create_transcription_table.sql
```

| Configuração | Objetivo |

---

## Isolamento arquitetural

Mesmo com a introdução da persistência, nenhuma regra de negócio foi movida para a camada de infraestrutura.

### Garantias mantidas

- Domínio continua puro
- JPA permanece restrito aos adapters
- Nenhuma entidade de domínio depende de Hibernate
- Infraestrutura permanece substituível

# Decisão de arquitetura

## Por que PostgreSQL?

PostgreSQL foi escolhido por oferecer:

- Confiabilidade
- Suporte robusto a transações
- Excelente integração com Hibernate
- Facilidade futura para índices, JSONB e busca textual

---

## Por que Flyway ao invés de geração automática do Hibernate?

Foi evitado o uso de `ddl-auto=create` como mecanismo principal de schema.

### Motivos

| Flyway | Hibernate Auto DDL |
|---|---|
| Histórico versionado | Sem rastreabilidade |
| Reproduzível | Dependente do estado atual |
| Seguro em produção | Alto risco |
| Compatível com CI/CD | Não recomendado para produção |

---

## Trade-offs adotados

| Decisão | Benefício | Trade-off |
|---|---|---|
| PostgreSQL dedicado | Persistência real | Maior consumo local |
| Flyway | Controle de schema | Necessidade de migrations |
| JPA/Hibernate | Produtividade | Complexidade ORM |
| Pool HikariCP | Performance | Configuração adicional |

# Troubleshooting

## Erro: Connection refused

### Causa

PostgreSQL não inicializou corretamente.

### Solução

```bash
docker compose logs postgres
```

Validar:

- Porta 5432 livre
- Container saudável
- Credenciais corretas

---

## Erro: Flyway validation failed

### Causa

Migration alterada após execução.

### Solução

Nunca editar migrations já executadas.

Criar nova migration incremental:

```text
V2__fix_table.sql
```

---

## Erro: Hibernate Dialect não encontrado

### Causa

Driver PostgreSQL ausente ou incompatível.

### Solução

Verificar dependência:

```gradle
runtimeOnly 'org.postgresql:postgresql'
```
