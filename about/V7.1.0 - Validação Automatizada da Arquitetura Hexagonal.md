# V7.1.0 — Validação Automatizada da Arquitetura Hexagonal

## Visão Geral

Esta etapa introduz governança arquitetural automatizada utilizando ArchUnit.

O objetivo da implementação é validar continuamente as regras da arquitetura hexagonal através de testes automatizados executados junto da suíte padrão do projeto, reduzindo risco de acoplamento indevido entre domínio, aplicação e infraestrutura.

A abordagem transforma convenções arquiteturais em regras executáveis garantindo maior previsibilidade na evolução do sistema.

---

# O que foi implementado

## Dependência ArchUnit

Foi adicionada a dependência:

```gradle
testImplementation 'com.tngtech.archunit:archunit-junit5:1.4.2'
```

## HexagonalArchitectureTest

Foi criada a suíte responsável pela validação arquitetural do projeto.

### Objetivos

| Validação | Objetivo |
|---|---|
| Onion Architecture | Isolamento entre camadas |
| Convenções de portas | Padronização arquitetural |
| Separação adapters | Evitar acoplamento |
| Restrições de dependência | Governança contínua |

---

## Onion Architecture

A arquitetura é validada automaticamente:

```java
@ArchTest
public static final ArchRule arquitetura_hexagonal_completa =
    onionArchitecture()
        .domainModels("..transcription.domain.model..")
        .applicationServices("..transcription.application..")
        .adapter("http", "..transcription.infrastructure.http..");
```

---

## Convenções de portas

Foi adicionada validação automática para nomenclatura de portas.

```java
@ArchTest
public static final ArchRule portas_de_entrada_devem_terminar_com_port =
    classes()
        .that()
        .resideInAPackage("..port.in..")
        .should()
        .haveSimpleNameEndingWith("Port");
```

# Decisão de arquitetura

## Por que ArchUnit?

ArchUnit permite transformar arquitetura em código verificável.

Isso reduz:

- erosão arquitetural
- acoplamento indevido
- violações silenciosas
- dependências incorretas

---

## Por que validar Onion Architecture?

A arquitetura hexagonal depende fortemente do isolamento entre camadas.

Sem validação automatizada, é fácil introduzir:

- adapters chamando domínio incorretamente
- infraestrutura vazando para aplicação
- regras técnicas dentro do domínio

---

## Por que validar convenções?

Padronização melhora:

- legibilidade
- onboarding
- previsibilidade
- manutenção futura

---

## Trade-offs adotados

| Decisão | Benefício | Trade-off |
|---|---|---|
| ArchUnit | Governança contínua | Mais testes |
| Onion validation | Segurança arquitetural | Rigidez estrutural |
| Convenções automáticas | Padronização | Menor flexibilidade |
| Testes arquiteturais | Evolução segura | Curva inicial |

# Troubleshooting

## Erro: Architecture Violation

### Causa

Dependência indevida entre camadas.

### Solução

Garantir:

```text
domain → sem infraestrutura
application → sem framework
```

---

## Erro: Package não encontrado

### Causa

Pacotes informados incorretamente no ArchUnit.

### Solução

Validar estrutura real dos packages.

---

## Erro: Testes arquiteturais falhando após refactor

### Causa

Mudança estrutural não refletida nas regras.

### Solução

Atualizar convenções da suíte arquitetural.
