# V9.1.0 — Domínio analysis/ + LanguageModelPort

## Visão Geral

Esta etapa introduz um novo bounded context completamente independente chamado `analysis/`, responsável pelo processamento semântico das transcrições utilizando modelos de linguagem.

A implementação mantém separação rigorosa entre os contextos de transcrição e análise, preservando baixo acoplamento e permitindo evolução independente dos providers de IA.

---

# O que foi implementado

## Novo bounded context analysis/

Foi criado um novo domínio separado do contexto `transcription/`.

### Objetivos

| Objetivo | Benefício |
|---|---|
| Separação semântica | Baixo acoplamento |
| Evolução independente | Escalabilidade |
| Regras próprias | Clareza arquitetural |
| Providers intercambiáveis | Flexibilidade |

---

## LanguageModelPort

Foi criada a porta responsável pela abstração do provider LLM.

```java
public interface LanguageModelPort {

    String generate(String prompt);
}
```

---

## SummarizeTranscriptionUseCase

Fluxo principal:

```text
1. Buscar transcrição
2. Gerar prompt
3. Chamar LLM
4. Persistir resumo
5. Retornar resposta
```

---

## Ollama

Foi utilizado provider local baseado em:

```text
qwen2.5:0.5b
```

ou:

```text
tinyllama
```

---

## Independência arquitetural

A troca futura para OpenAI, Claude ou Gemini exigirá apenas novo adapter.

O domínio permanece inalterado.

# Decisão de arquitetura

## Por que novo bounded context?

Análise semântica representa responsabilidade distinta de transcrição.

Separar evita:

- domínio inchado
- mistura semântica
- acoplamento funcional

---

## Por que LanguageModelPort?

Permite troca transparente de providers LLM.

---

## Trade-offs adotados

| Decisão | Benefício | Trade-off |
|---|---|---|
| Novo contexto | Organização clara | Mais estrutura |
| Porta LLM | Flexibilidade | Mais abstração |
| Ollama local | Sem custo variável | Consumo local |
| Modelo ≤1B | Baixo hardware | Menor qualidade |
