# V11.5.0 — Sistema de Notificações Multicanal

## Visão Geral

Esta fase introduz um novo bounded context chamado `notification`, responsável pelo envio de notificações orientadas a eventos.

A implementação foi construída sobre a infraestrutura de mensageria introduzida na Fase 10, utilizando RabbitMQ como mecanismo de integração entre domínios.

O objetivo é permitir que diferentes canais de comunicação sejam adicionados sem alterar regras existentes, preservando os princípios da Arquitetura Hexagonal e do Open/Closed Principle.

---

# O que foi implementado

## Novo bounded context notification

Foi criado um domínio completamente independente do contexto `transcription`.

Estrutura:

```text
notification
├── domain
├── application
├── infrastructure
│   ├── messaging
│   ├── email
│   ├── sms
│   └── whatsapp
```

---

## Modelo de domínio Notification

Representação padronizada utilizada por todos os canais.

```java
public record Notification(
    String recipient,
    String subject,
    String message
) {}
```

### Objetivos

| Objetivo               | Benefício         |
| ---------------------- | ----------------- |
| Modelo único           | Reutilização      |
| Independência de canal | Baixo acoplamento |
| Evolução simplificada  | Extensibilidade   |

---

## SendNotificationPort

Porta principal de saída do domínio.

```java
public interface SendNotificationPort {

    void send(Notification notification);

}
```

Permite que a aplicação solicite envios sem conhecer detalhes de SMTP, SMS ou APIs externas.

---

## Consumo de eventos RabbitMQ

Foi implementado adapter responsável por consumir eventos publicados pela aplicação.

```java
@RabbitListener(
    queues = "notification.queue"
)
public void consume(NotificationEvent event) {

    sendNotificationUseCase.execute(event);

}
```

# Decisão de arquitetura

## Por que um bounded context separado?

Notificação representa uma responsabilidade diferente da transcrição.

Sem separação:

```text
transcription
 ├── envio email
 ├── envio sms
 ├── whatsapp
 └── regras de transcrição
```

Com bounded context:

```text
transcription
        ↓
       evento
        ↓
notification
```

---

## Por que consumir eventos?

Evita dependência direta entre domínios.

O contexto de transcrição não conhece:

* Email
* SMS
* WhatsApp
* Provedores externos

---

## Por que múltiplas portas?

Cada canal possui características próprias.

Permite:

* Troca de provider
* Evolução independente
* Testes isolados
* Configuração por ambiente

---

## Trade-offs

| Decisão            | Benefício                 | Trade-off                      |
| ------------------ | ------------------------- | ------------------------------ |
| RabbitMQ           | Desacoplamento            | Mais infraestrutura            |
| Bounded Context    | Organização clara         | Mais componentes               |
| Multi-canal        | Flexibilidade             | Mais adapters                  |
| Portas específicas | Independência tecnológica | Mais abstrações                |
| Eventos            | Escalabilidade            | Maior complexidade operacional |

# Troubleshooting

## Evento não consumido

### Causa

Fila incorreta.

### Solução

Validar binding da queue.

---

## Canal não executa

### Causa

Adapter não registrado.

### Solução

Verificar configuração Spring.

---

## Mensagem vai para DLQ

### Causa

Falha no processamento.

### Solução

Verificar logs do consumer.
