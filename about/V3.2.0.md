# v3.2.0 — Entidade de Domínio e Portas de Persistência

## Visão Geral

Esta etapa introduz a modelagem inicial do domínio de persistência das transcrições dentro da arquitetura hexagonal.

Foi criada a entidade de domínio `Transcription` juntamente das portas de saída responsáveis por abstrair persistência e consulta por hash de áudio.

A implementação mantém o domínio completamente desacoplado de JPA, Hibernate, PostgreSQL ou qualquer detalhe de infraestrutura.

O objetivo principal desta etapa é preparar a aplicação para reutilização inteligente de transcrições futuras através de deduplicação por hash SHA-256.

---

# O que foi implementado

## Entidade de domínio `Transcription`

Foi criada a entidade responsável por representar uma transcrição persistida no contexto da aplicação.

### Responsabilidades

| Campo | Objetivo |
|---|---|
| id | Identificador único UUID |
| audioHash | Hash SHA-256 do áudio |
| content | Conteúdo transcrito |
| createdAt | Timestamp de criação |

---

## Características da entidade

A entidade foi modelada como objeto puro de domínio.

### Garantias arquiteturais

- Sem dependência de Hibernate
- Sem annotations JPA
- Sem dependência Spring
- Sem acoplamento com banco
- Sem serialização HTTP

---

## Portas de persistência

Foi criada a abstração responsável por comunicação entre domínio e infraestrutura.

### Interface principal

```java
public interface TranscriptionGateway {

    Optional<Transcription> findByAudioHash(String hash);

    Transcription save(Transcription transcription);
}
```

---

## Objetivo das portas

As portas permitem que o domínio:

- Consulte transcrições existentes
- Salve novas transcrições
- Permaneça independente da infraestrutura
- Troque mecanismos de persistência futuramente

---

## Estratégia de deduplicação futura

O hash SHA-256 do áudio foi introduzido como identificador lógico de conteúdo.

### Benefícios futuros

| Benefício | Objetivo |
|---|---|
| Reuso inteligente | Evitar retranscrição |
| Economia computacional | Reduzir processamento Whisper |
| Escalabilidade | Menor custo de inferência |
| Cache futuro | Integração futura com Redis |

Nesta etapa ainda não existe implementação concreta de cache ou persistência.

Apenas a modelagem arquitetural foi preparada.

# Decisão de arquitetura

## Por que a entidade não utiliza JPA?

A entidade de domínio foi mantida pura para preservar os princípios da arquitetura hexagonal.

### Benefícios

| Domínio puro | Entidade JPA acoplada |
|---|---|
| Independente de framework | Dependente Hibernate |
| Fácil de testar | Mais difícil isolar |
| Reutilizável | Acoplada infraestrutura |
| Portável | Limitada ORM |

---

## Por que usar portas ao invés de repositories Spring diretamente?

O domínio não deve conhecer tecnologias de persistência.

As portas criam uma fronteira arquitetural explícita.

### Resultado

```text
Domain
  └── Port (SPI)
          └── Adapter
                  └── JPA
                          └── PostgreSQL
```

---

## Trade-offs adotados

| Decisão | Benefício | Trade-off |
|---|---|---|
| Entidade pura | Baixo acoplamento | Conversão futura necessária |
| Porta SPI | Flexibilidade | Mais interfaces |
| Hash SHA-256 | Deduplicação futura | Custo de cálculo |
| UUID | Segurança e rastreabilidade | Índices maiores |

# Troubleshooting

## Erro: Dependência circular entre domínio e infraestrutura

### Causa

Importação incorreta de classes JPA dentro do domínio.

### Solução

Garantir:

```text
domain -> NÃO conhece adapters
adapters -> conhecem domínio
```

---

## Erro: Optional não resolvido

### Causa

Import ausente.

### Solução

```java
import java.util.Optional;
```

---

## Erro: UUID incompatível

### Causa

Uso inconsistente entre String e UUID.

### Solução

Padronizar:

```java
UUID id
```

Evitar uso textual no domínio.
