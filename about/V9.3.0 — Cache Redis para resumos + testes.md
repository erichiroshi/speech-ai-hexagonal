# V9.3.0 — Cache Redis para resumos + testes

## Visão Geral

Esta etapa introduz cache distribuído dedicado para resumos gerados pelos modelos de linguagem.

O objetivo é evitar reprocessamento desnecessário do LLM reduzindo latência e consumo computacional.

---

# O que foi implementado

## SummaryStorePort

Nova porta de saída responsável pelo armazenamento de resumos.

### Responsabilidades

| Responsabilidade | Objetivo |
|---|---|
| Salvar resumo | Reutilização |
| Buscar resumo | Cache hit |
| Aplicar TTL | Expiração automática |

---

## RedisStoreAdapter

Estratégia de chave:

```text
summary:{hash}
```

---

## TTL configurável

TTL padrão:

```text
72 horas
```

---

## Fluxo de cache

```text
1ª chamada:
LLM → Redis → cached=false

2ª chamada:
Redis hit → cached=true
```

---

## RedisTemplate separado

Foi criada configuração específica para resumos.

### Objetivos

- isolamento lógico
- serialização dedicada
- evolução independente

---

## Testcontainers

Testes reais validam:

- cache hit
- cache miss
- TTL
- integração Redis

# Decisão de arquitetura

## Por que cache separado?

Resumo possui comportamento diferente de transcrição.

Separar permite:

- TTL independente
- serialização específica
- métricas próprias

---

## Por que cachear resumos?

LLMs possuem alto custo computacional comparado ao Redis.

---

## Trade-offs adotados

| Decisão | Benefício | Trade-off |
|---|---|---|
| Cache Redis | Baixa latência | Mais memória |
| TTL 72h | Reuso eficiente | Dados temporários |
| RedisTemplate separado | Isolamento | Mais configuração |
| Testcontainers | Validação real | Execução mais lenta |

---

A aplicação agora possui:

- Cache distribuído de resumos
